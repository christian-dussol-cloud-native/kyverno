# Kyverno Annotation Standards

Reference for proper Kyverno policy annotations following official conventions.
Source: https://github.com/kyverno/policies (contribution guidelines)

## Required Annotations

Every policy MUST include these annotations:

```yaml
annotations:
  # Short, descriptive title
  policies.kyverno.io/title: "Require Resource Limits"

  # Category for filtering on kyverno.io/policies
  policies.kyverno.io/category: "Best Practices"

  # Impact severity: low | medium | high | critical
  policies.kyverno.io/severity: "medium"

  # Multi-line description explaining what and why
  policies.kyverno.io/description: >-
    Requires all containers to specify CPU and memory resource
    requests and limits. This ensures proper scheduling and
    prevents resource starvation in shared clusters.
```

## Recommended Annotations

```yaml
annotations:
  # Primary Kubernetes resource the policy targets
  policies.kyverno.io/subject: "Pod"

  # Minimum Kyverno version required
  policies.kyverno.io/minversion: "1.6.0"
```

## Standard Categories

Use these official categories for consistency with kyverno.io/policies:

| Category | When to use |
|----------|------------|
| Best Practices | General K8s best practices |
| Pod Security | PSA baseline/restricted controls |
| Resource Management | CPU, memory, quotas, limits |
| Networking | NetworkPolicy, Services, Ingress |
| Supply Chain Security | Image verification, signatures |
| RBAC | Roles, bindings, ServiceAccounts |
| Multi-Tenancy | Namespace isolation, labels |
| Other | Custom/uncategorized |

### Custom categories (Kyverno Skills originals):

| Category | When to use |
|----------|------------|
| FinOps | Cost optimization, chargeback, budgets |
| FinOps, Cost Optimization | Combined FinOps + cost focus |
| FinOps, Resource Management | FinOps + resource limits |
| Compliance | Regulatory frameworks (CIS, NIST, PCI) |

## Severity Guidelines

| Severity | When to use | Examples |
|----------|------------|---------|
| low | Nice-to-have, minor impact | Default labels, naming conventions |
| medium | Recommended standard | Resource limits, probes, image tags |
| high | Security or cost risk | Privileged containers, missing quotas |
| critical | Immediate risk, must enforce | Root access, cluster-admin binding, unsigned images |

## Labels Convention

Kyverno Skills adds these labels for management:

```yaml
labels:
  # Identifies the policy was generated by Kyverno Skills
  app.kubernetes.io/managed-by: kyverno-skills

  # Policy rule type for filtering
  kyverno-skills/type: "validate"  # validate | mutate | generate | verifyImages

  # Optional: source skill that generated it
  kyverno-skills/source: "kyverno-policy-generator"
```

## String Value Conventions

- String values do NOT need quotes in Kyverno policies
- Exception: JMESPath expressions `{{request.operation}}` need to be in strings
- Pattern anchors: `"?*"` means "must exist and be non-empty"
- Enum patterns: `"dev | qa | staging | prod"` means one of these values
- Negation: `"!*:latest"` means "must NOT match this pattern"
- Conditional anchors: `+(key)` in mutate means "add only if absent"
